    🧪 Frontend System Design Prompt (Banking Project)

    📝 Prompt:

    "Design the frontend architecture for a web banking application used by millions of users. The app includes features like viewing account balances, transferring money, transaction history, user settings, and support chat."

    🧠 Step-by-Step: How to Think Like an Architect

    ✅ 1. Requirements Clarification (Always Start Here)

    Ask clarifying questions:

    • Users: Retail customers only? Or include business accounts?
    • Devices: Web + mobile web? Responsive?
    • Data freshness: Do balances update in real-time?
    • Security: How strict is the session/token handling?
    • Scale: Millions of users? Any internationalization?
    • Feature scope: MVP or long-term design?

    🧱 2. High-Level Architecture

    Banking Web App
├── Auth Layer (SSO, MFA, token refresh)
├── Core Layout (NavBar, Sidebar, Footer)
├── Modules
│   ├── Accounts
│   ├── Transactions
│   ├── Transfers
│   ├── Settings
│   └── Support (chat, helpdesk)

    Use a feature-based folder structure:

    /src
  /features
    /accounts
    /transactions
    /transfers
    /settings
    /support
  /components
    /ui
    /layout
  /lib
  /hooks
  /services
  /state
  /types
  /pages (if using Next.js)

    🧠 3. Key Architectural Considerations

    🔐 Authentication & Authorization

    • Use JWT or session cookies (secure, HTTP-only)
    • Multi-factor auth (MFA) integration
    • Role-based access control (RBAC) on frontend routes + UI

    🧾 State Management

    Type Tool
    Server state React Query (TanStack Query)
    Client state Zustand / Redux Toolkit
    Form state React Hook Form + Zod/Yup
    • Use query caching to minimize balance fetches
    • Use React Query for transactions, with pagination + prefetching

    🧩 Component Architecture

    • Atomic design or headless components
    • Shared design tokens (spacing, color, typography)
    • Accessible UI (ARIA, keyboard nav, screen reader)

    🧪 Testing Strategy

    • Jest for unit tests
    • React Testing Library for component tests
    • Cypress or Playwright for E2E testing

    Test scenarios:

    • Token expiry
    • Large transaction list
    • Concurrent transfer edge cases

    🌍 Internationalization (i18n)

    • Currency, language, locale formatting
    • Use i18next or react-intl

    🧰 Error & Logging

    • Use Error Boundaries
    • Connect to Sentry for error tracking
    • Fallback UIs for network/API errors

    🚀 4. Performance & Scalability

    Concern Strategy
    Bundle size Code splitting with dynamic imports
    Large data sets Virtualize long lists (e.g., react-window)
    API optimization Pagination, optimistic updates, background refetch
    Rendering SSR/CSR with Next.js (Next 13 app router)
    Caching HTTP + React Query + CDN for static assets

    ⚙️ 5. DevOps & CI/CD Considerations

    • Linting, Prettier, and TypeScript strict mode
    • GitHub Actions / GitLab CI
    • Feature flags for rolling out features (LaunchDarkly or homegrown)
    • Lighthouse CI to catch regressions
    • End-to-end monitoring

    🔒 6. Security First (Critical for Banking)

    • Sanitize all inputs (forms, chat)

    • Use Content Security Policy (CSP)

    • Prevent:

      • XSS
      • CSRF (use proper tokens)
      • Clickjacking (use frame options headers)

    • Set up proper error logging without leaking sensitive data

    🧠 7. Think Like a Team Lead

    • Separate modules for each major feature (accounts, transfers, etc.)
    • Define shared interfaces and contracts between features
    • Document component APIs and expected behaviors
    • Use a design system to unify look/feel across modules
    • Consider monorepo if building multiple apps (e.g., admin portal, customer app)

    🏁 TL;DR – Final Output Summary

    Area Your Choices
    Folder Structure Feature-based (scalable)
    State Management React Query + Zustand/Redux
    Component Architecture Headless + atomic + accessible
    Performance Code-splitting, caching, virtualization
    Testing Jest + RTL + Cypress
    Auth JWT + MFA + RBAC
    DevOps CI/CD, feature flags, monitoring
    Security XSS/CSRF/CSP + error sanitization
    Design System Custom tokens + Storybook
    Team Scaling Shared contracts, documentation, module separation

├── 📁 features/

│   │   ├── components/
│   │   ├── pages/
│   │   └── hooks/
│   ├── 📁 transactions/    → History, filters, pagination
│   ├── 📁 transfers/       → Send money, confirmation flow
│   ├── 📁 settings/        → Profile, MFA, language
│   └── 📁 support/         → Chat, contact, FAQs
│
├── 📁 components/          → Reusable UI components (buttons, modals)
│   ├── ui/                 → Generic, styled (design system)
│   ├── layout/             → Navbar, Sidebar, Page shell

│
├── 📁 lib/                 → Utility functions (formatting, helpers)

│   └── useResponsive.ts
│
├── 📁 services/            → API layer
│   ├── accountService.ts
│   ├── transactionService.ts
│   └── transferService.ts
│
├── 📁 state/               → Global state
│   ├── auth.store.ts       → Zustand or Redux slices
│   ├── ui.store.ts         → Modals, toasts, theme
│   └── transfer.store.ts
│
├── 📁 types/               → Global TS types/interfaces
│   ├── account.ts

├── 📁 i18n/                → Translations, i18n setup
│   └── en/

├── 📁 tests/               → E2E, integration tests
│   └── transactions.spec.ts
│
├── 📄 index.tsx
├── 📄 App.tsx
└── 📄 main.tsx (if using Vite)

    🧠 Layered View (Abstracted Architecture)

    [Presentation Layer]        ─── Reusable UI components, feature UIs
     ↓
[Page & Layout Layer]       ─── Routing, App Shell, Protected Routes
     ↓
[Feature Modules Layer]     ─── Business logic per domain (Accounts, Transfers)
     ↓
[Hooks / State Layer]       ─── Shared state (Zustand), reusable hooks
     ↓
[Services / API Layer]      ─── Typed API clients, error handling
     ↓
[Utility Layer]             ─── Formatters, date/currency helpers

    ⚙️ Tooling & Systems Represented

    Concern Solution in Diagram
    Modularity features/ and components/ folders
    State Mgmt state/ (Zustand/Redux per domain)
    Reusability components/ui/ and hooks/
    API Calls services/ and lib/api-client.ts abstraction
    Global Types types/ shared between services, components
    I18n i18n/ using i18next or similar
    Testing tests/ folder with Cypress or Playwright
    Performance Lazy imports in features/, React.Suspense
    Design System components/ui/, styled with Tailwind/CSS-in-JS
    Error Handling Centralized in api-client.ts and Error Boundaries

    🧩 BONUS: Feature Module Breakdown Example – /features/transfers/

    /transfers
├── components/
│   ├── TransferForm.tsx        → Handles form UI
│   └── TransferConfirmation.tsx
├── pages/
│   └── TransferPage.tsx        → Route-level page for transfers
├── hooks/
│   └── useTransferFlow.ts      → Encapsulates logic across steps
├── api/
│   └── transfer.api.ts         → Fetch/send transfer data
├── store/
│   └── transfer.store.ts       → Local state for transfer steps
├── types/
│   └── transfer.ts             → Request/response interfaces

    🎯 Summary

    This structure:

    • Scales across teams (each feature has its own space)
    • Keeps logic local to features
    • Supports global/shared logic via components/, hooks/, state/, and lib/
    • Encourages testing, performance, and security best practices
    • Makes onboarding new devs easier

    🛡️ Security Layer Summary

    Component Security Best Practices
    React Use HTTPS, never store LDAP passwords
    Backend Input validation, secure bind to LDAP, use rate limiting
    LDAP Use ldaps://, lock down network access, enable logging
    Tokens Use short-lived JWTs, secure cookies if needed
